# Code Review: Mirroring Logic Bug (Full Artist Mirroring)

## Issue Overview
When adding a single track or album to a list that requires mirroring (Scenario B/C), the background backfill process currently mirrors **all music** by that artist in the Lidarr library, rather than just the specific files requested.

This results in:
1.  **Duplicate/Incorrect Labeling**: Every file by the artist is copied and associated with the single `list_item_id` in the `mirror_files` table. The UI then shows many different files (e.g., entire albums) as being "T.N.T. by AC/DC".
2.  **Storage Inefficiency**: If a user adds one song from an artist with a large library, TuneFetch will attempt to copy hundreds of files.

## Technical Root Cause
The bug is located in `src/lib/server/mirror.ts` within the `_runBackfill` function.

```typescript
// src/lib/server/mirror.ts

async function _runBackfill(
  lidarrArtistId: number,
  listItemId: number,
  ownerRoot: string,
  targetRoot: string
): Promise<void> {
  // ...
  // BUG: This fetches EVERY file for the artist
  const trackFiles = await getTrackFiles(lidarrArtistId);
  
  // ...
  for (const file of trackFiles) {
    try {
      // BUG: Every file found for the artist is mirrored and 
      // associated with the trigger listItemId.
      await mirrorTrackFile(file.path, listItemId, ownerRoot, targetRoot);
      successCount++;
    } catch (err) {
      // ...
    }
  }
}
```

The function `getTrackFiles(lidarrArtistId)` is correctly named—it gets all files for that artist—but `_runBackfill` uses it without any filtering based on the `list_item`'s actual type (track/album/artist) or its specific Lidarr IDs (`lidarr_track_id` / `lidarr_album_id`).

## Confirmation of Reported Behavior
The user's reported logs confirm this:
-   `T.N.T. AC/DC` (a single track) is associated with tracks from *Highway to Hell*, *Back in Black*, and *Power Up*.
-   `Whatever It Takes Imagine Dragons` is associated with almost their entire discography (*Night Visions*, *Smoke + Mirrors*, *Mercury*, etc.).

## Recommended Fix
`_runBackfill` should be updated to fetch the `list_item` record and filter the `trackFiles` list accordingly.

### Proposed Logic Change:
1.  Fetch the `list_item` from the database using `listItemId`.
2.  Retrieve all `trackFiles` for the artist from Lidarr (as it does now).
3.  Filter `trackFiles` before the loop:
    -   If `listItem.type === 'artist'`: Mirror everything (existing behavior).
    -   If `listItem.type === 'album'`: Only mirror files where `file.albumId === listItem.lidarr_album_id`.
    -   If `listItem.type === 'track'`: Only mirror files that map to `listItem.lidarr_track_id`.

> [!NOTE]
> Lidarr's `LidarrTrackFile` response in `getTrackFiles` includes an `albumId`. For individual tracks, we may need to check the `tracks` array within the Lidarr response (not currently explicitly typed in `LidarrTrackFile` but usually present in the API) or match the track file to the track via a lookup.

## Conclusion
This is a **Confirmed Bug**. It is not expected behavior for a single track add to trigger a full artist mirror.
